# 算法接入：图像收口与视频下一版清单

## 1. 当前已完成口径（图像）

本轮完成的是“图像算法真实接入”的第一版，目标不是放开任意代码即时执行，而是做受控接入。

### 1.1 当前治理规则

- 用户先在“算法接入”提交代码包，由管理员审核。
- 审核通过后，只会生成“当前用户自己的 `UserPackage` 算法”，不会自动进入平台算法库。
- 管理员审核时可单独勾选“接入运行链路 / `runtime_ready`”。
- 只有审核通过、`runtime_ready=true`、算法处于可运行状态的 `UserPackage`，才允许在 `NewRun` 中被选择。
- 平台共享仍然通过管理员单独“收录为平台算法”完成，与“本人可运行”分离。

### 1.2 当前固定协议

第一版仅支持图像任务，不支持 `video_denoise` / `video_sr`。

执行命令固定为：

```bash
python infer.py --input <input_image> --output <output_image>
```

上传包支持两种形式：

- 单个 `.py` 文件：直接作为入口脚本执行。
- zip 包：优先寻找 `infer.py`；若压缩包内只有一个 `.py` 文件，则使用该文件。

### 1.3 当前后端执行方式

- 后端不 `import` 用户代码，而是由 Worker 使用 `subprocess` 隔离执行。
- Worker 会将输入图像写入临时目录，并要求脚本输出结果图像到指定路径。
- 执行完成后，系统读取输出图像，再接入现有 PSNR/SSIM/NIQE 与自定义指标链路。
- 若出现算法包缺失、入口脚本缺失、脚本超时、脚本返回非零、输出文件不可读等情况，Run 会失败并写入明确错误信息。

### 1.4 当前联动口径

- “算法接入”中的审核通过记录，会对应到“算法库 - 用户算法”中的一条私有 `UserPackage` 记录。
- 两处以同一条 submission 为源头进行同步，不应出现一边存在、一边不存在的长期漂移状态。
- 删除任一侧记录时，会同步清理关联的另一侧私有记录；若已有平台副本，不会误删平台算法。

## 2. 当前验收建议（图像）

建议至少完成以下 3 步：

1. 使用 `docs/项目资料/test_algorithm_denoise.py` 或同协议脚本，提交算法接入申请。
2. 管理员审核通过并勾选“接入运行链路”，确认“算法接入”与“算法库 - 用户算法”均出现该条记录。
3. 在 `NewRun` 选择本人已接入算法与真实图像数据集，创建 Run，并确认：
   - Run 状态为 `done`
   - 指标结果为真实计算值
   - 详情页中存在 `record.algorithm_runtime`

## 3. 下一版目标（视频算法接入）

下一版建议继续沿用当前“受控接入”的思路，把能力从图像扩展到视频，而不是另起一套体系。

### 3.1 建议保持不变的原则

- 继续使用管理员审核 + `runtime_ready` 控制。
- 继续使用 `subprocess` 隔离执行，不直接 `import` 用户代码。
- 继续在 Worker 中完成临时文件写入、超时控制、失败信息归一化。
- 继续保持“本人可运行”和“平台共享”分离。

### 3.2 视频下一版最小实现清单

1. 扩展协议到视频任务：
   - `python infer.py --input <input_video> --output <output_video>`
2. Worker 支持视频输入输出临时文件管理。
3. 将 `video_denoise` / `video_sr` 的 `UserPackage` 纳入 `NewRun` 可选范围。
4. 输出视频合法性校验：
   - 文件存在
   - 可读
   - 基本帧数/尺寸满足评测要求
5. 复用现有视频指标链路，保证 `PSNR/SSIM/NIQE` 与导出一致。
6. 补齐失败路径：
   - 包缺失
   - 入口脚本不存在
   - 超时
   - 输出视频不可读
   - 帧数或尺寸不匹配

## 4. 完成度判断建议

若以下三项均稳定成立，可认为“算法接入”模块已经完成图像版闭环：

- 审核通过后，本人算法可见、可管、可删、可真实运行。
- 平台算法收录动作独立，不会因为审核通过而自动全站开放。
- Run 详情能证明系统真实执行了上传脚本，而不是仍停留在基线分支或模拟分支。
