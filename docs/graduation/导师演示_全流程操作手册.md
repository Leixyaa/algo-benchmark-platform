# 导师演示全流程操作手册（防翻车版）

## 0. 开场前 1 分钟

- 在**项目根目录**打开终端，执行：`powershell -ExecutionPolicy Bypass -File .\scripts\demo_preflight_check.ps1`
- **新环境**须先在 `web/` 下执行 `npm install`，再 `npm run dev`（仓库根目录不再有 `package.json` / `node_modules`）。
- 自检通过后再开始演示。
- 若 AI key 未配置，不阻塞主流程；AI 客服会走本地兜底回复。
- 打开 `docs/graduation/演示数据固化清单.md`，确认图像/视频固定数据集与算法没有变更。

## 1. 推荐演示账号

- 普通用户：用于展示“业务闭环”
- 管理员：用于展示“审核治理”

说明：先用普通用户完整走一遍，再切管理员做治理动作，避免来回切账号影响节奏。

## 2. 演示顺序（固定）

1. `数据集管理`
2. `算法库`
3. `发起评测`
4. `任务中心`
5. `结果对比`
6. `我的通知`（右上角）
7. `个人信息`（右上角下拉）
8. `AI 客服`（可选，1 分钟内）

（可选加一步：`社区中心`，与数据集/算法联动时演示。）

## 3. 每步点击与通过标准

### 3.1 数据集管理

- 操作：选择一套已准备的数据集；管理员可演示「扫描」。新建数据集时**不强制**用户手填任务类型；发布到社区前系统会校验**已识别任务**（扫描/导入后的 `pairs_by_task`、`task_types` / `supported_task_types` 等元数据）。
- 通过标准：在**当前演示任务类型**下，可用配对数 `> 0`（`发起评测` / `结果对比` 中数据集列表会按任务类型过滤，只显示支持该任务的数据集）。
- 兜底说法：若配对为 0，说明输入/GT 目录不匹配，系统会阻止创建任务，这是保护机制。

### 3.2 算法库

- 操作：确认至少 1 个可用算法（建议准备 2～3 个方便对比）。算法接入说明可通过 `/algorithm-access` 跳转至本页对话框。
- 通过标准：算法可见、状态正常、可被发起评测页面选择。

### 3.3 发起评测

- 操作：**两步**：① 基础配置（任务类型、数据集、评测模式、可选历史预设）→ ② 算法选型（多选算法、指标、可选「保存预设」支持**多算法一并保存**）→「启动评测任务」。
- 通过标准：弹出创建成功提示，可跳转任务中心。
- 关键口径：**无「参数方案」下拉**；运行参数统一为各算法**默认参数**；预设可保存当前任务/数据集/指标与所选算法集合。

### 3.4 任务中心

- 操作：查看新任务，打开详情。
- 通过标准：看到状态与时间链路（创建/开始/完成），可见指标值。
- 关键口径：列表操作为「**删除**」且**真实落库**，不是仅本地隐藏。

### 3.5 结果对比

- 操作：选择任务和数据集，查看排序结果。
- 通过标准：能看到真实结果条数、当前最优、并可导出 CSV/Excel/Markdown。
- 关键口径：筛选会按任务过滤数据集；过期本地缓存会自动规范化，不残留无效选择。

### 3.6 通知与个人信息

- 操作：
  - 右上角进入「我的通知」，演示未读/已读筛选、全部已读、清理已读
  - 右上角下拉进入「个人信息」，修改显示名称（不改用户名）
- 通过标准：操作后界面状态即时更新。

### 3.7 AI 客服（可选）

- 操作：问「怎么发起评测」「算法接入步骤」
- 通过标准：
  - 配置模型时：返回模型回答
  - 未配置模型时：返回本地兜底，不报错中断

## 4. 现场禁忌（防翻车）

- 不临时改环境变量、不过度改代码。
- 不在现场演示「首次接入新模型」。
- 不演示未准备的数据集或未扫描的数据集。
- 不临时切换到未知账号。

## 5. 出现异常时的固定话术

- 「这里是系统的保护校验，避免错误数据进入评测流程。」
- 「我先切到已准备数据继续演示主链路，保证验收节奏。」
- 「这一步不是功能缺失，是配置项未启用；系统提供了离线兜底能力。」

## 6. 演示结束收尾话术

- 「平台已形成数据、算法、评测、对比、治理、通知与个人信息的闭环。」
- 「主流程可复现，异常有兜底，导出与追溯能力完整。」

## 附录 A：环境与启动（与当前仓库一致）

1. 项目根目录示例：`d:\algo-benchmark-platform`（按本机修改）。
2. 后端（端口以本机为准，演示自检脚本默认检测 **8001** 时请与启动端口一致或改脚本参数）：
   - `cd backend` 后激活 venv，执行 `python -m uvicorn app.main:app --host 127.0.0.1 --port 8001`
3. Celery Worker：按你方既有方式启动 `backend/app/tasks.py` 对应 worker。
4. 前端：
   - `cd web`
   - **首次或新环境**：`npm install`
   - `npm run dev`
5. 浏览器打开前端地址；另开 `http://127.0.0.1:<端口>/docs` 确认 API 可用。
6. **说明**：终端里用 Ctrl+C 停止 `uvicorn` 时出现 `KeyboardInterrupt` / `CancelledError` 为**正常关机现象**，不是业务 Bug。

## 附录 B：数据集 / 任务类型 / 社区（实现要点，便于答辩）

- 前端工具：`web/src/utils/datasetTask.js`（`datasetSupportsTaskType`、`datasetHasRecognizedTasks` 等）。
- 扫描/导入后，后端会把 `meta.supported_task_types` 与 `task_types` 对齐；列表与详情使用 `pairs_by_task` 等字段。
- 社区数据集列表：`GET /datasets?scope=community` 排除 `owner_id=system`，并对候选集再 `load_dataset` 校验公开可用。
- **社区中心**页面不纳入 `LayoutBrand2` 的 `keep-alive` 缓存，避免离开再进仍显示旧列表快照。

## 附录 C：存储与 MySQL（答辩一句话）

- Redis 与 SQL 双写时，加载与列表合并**取较新**；`init_schema` 对 DDL 加锁并重试 MySQL **1683/1684** 类错误。
- 详见：`backend/app/store.py`、`backend/app/sql_store.py`。

## 附录 D：详细点击流（可选展开）

### D.1 数据集管理（管理员扫描）

1. 左侧「数据集管理」→ 选择已准备的数据集 →「扫描当前数据集」（若需）。
2. 确认在固化任务类型下 `pairs_by_task` 对应项 `> 0`。

### D.2 发起评测（两步）

1. **步骤一**：任务类型、数据集、评测模式（预览/全量）、可选历史预设。
2. **步骤二**：多选算法、勾选指标（内置必选）、可选填写预设名称并「保存预设」（可多算法）、「启动评测任务」。
3. 无「参数方案」步骤与下拉；参数为各算法默认。

### D.3 任务中心

- 使用「删除」移除记录（后端真实删除）；「取消」用于运行中任务。

### D.4 结果对比

- 选择任务后，数据集下拉仅展示**支持该任务**的数据集；导出与页面一致即可。
