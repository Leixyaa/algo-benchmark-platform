# 全功能测试用例（稳定性回归版）

## 1. 文档目的
- 适用范围：`algo-benchmark-platform` 当前代码口径下的前端页面、后端 API、异步执行链路、社区能力、管理端治理能力、导出能力与 MySQL 迁移后的持久化能力。
- 使用目标：把“能不能用”和“稳不稳定”分开验证，覆盖主流程、权限、异常、审核、导出、社区联动和资源清理等边界能力。
- 版本口径：按 2026-04-12 仓库现状整理；不再把旧版“智能快速选型入口”作为当前必测项，Compare 页以当前“筛选 + 排序 + 推荐结论导出”实现为准。

## 2. 覆盖模块
- 账号与权限
- 数据集管理
- 算法接入与算法资源管理
- 指标库
- 参数预设
- Run 执行与任务中心
- Compare 对比与导出
- 社区中心
- 管理员后台
- 稳定性、真实性与异常回归
- MySQL 迁移后的存储一致性

## 3. 执行前准备

### 3.1 环境
- 在项目根目录执行 `./scripts/manual_up.cmd` 或 `powershell -ExecutionPolicy Bypass -File .\scripts\manual_up.ps1 -WorkerCount 4`。
- 确认后端 Swagger 可访问：`http://127.0.0.1:8000/docs`。
- 确认前端页面可访问：`http://localhost:5173/`。
- 如果当前脚本使用其他端口，请以实际启动日志为准。

### 3.2 测试账号

| 角色 | 账号 | 用途 |
|---|---|---|
| 管理员 | `admin` / `Admin@123456` 或环境变量覆盖值 | 管理端审核、下架、举报处理、收录平台算法 |
| 普通用户 A | 自注册 | 创建数据集、提交算法接入、发布社区资源、创建 Run |
| 普通用户 B | 自注册 | 验证跨账号权限、社区下载、评论、举报、下架联动 |

### 3.4 证据留存要求
- 每条用例至少留 1 份证据：页面截图、导出文件、Swagger 响应或日志截图三选一。
- 与异步任务相关的用例，额外保留任务状态变化截图。
- 与审核、举报、下架相关的用例，额外保留“处理前”和“处理后”两张图。

## 4. 通过判定规则
- 单条用例结论仅记录“通过 / 失败 / 阻塞”。
- 主链路模块通过率目标：`>= 95%`。
- 稳定性模块通过标准：连续批量创建 10+ Run 后无后端崩溃、无大面积 stuck；取消状态能收敛；导出文件可打开；关键错误均返回可读提示或结构化错误码。

| ADM-04 | 算法接入审核与日志 | 已有待审核接入申请 | 管理员审核后查看“算法接入日志” | 处理记录完整 | 审核说明与运行接入状态一致 |

## 10. 最终补充专项（迁移后必测）

### 10.1 迁移后存储一致性专项

| 编号 | 场景 | 前置条件 | 操作步骤 | 预期结果 | 稳定性关注点 |
|---|---|---|---|---|---|
| MIG-01 | 数据集记录与磁盘目录一致 | 已有 1 个真实数据集 | 新建/导入数据集后，记录 `dataset_id` 与 `storage_path`，检查磁盘目录，再执行扫描与下载 | 页面记录、扫描结果、下载内容与磁盘目录一致 | 不出现“页面有记录但目录不存在” |
| MIG-02 | ZIP 导入后的持久化一致 | 准备合法 ZIP | 导入 ZIP -> 扫描 -> 下载到本地 -> 重启后再次查看 | `storage_path`、配对数、下载内容、重启后列表状态一致 | 避免 MySQL 记录与本地文件脱节 |
| MIG-03 | 结果页与任务中心口径一致 | 已存在多条完成 Run | 分别进入任务中心、结果对比、导出页面查看同一批次结果 | 列表数量、核心指标、导出口径一致 | 避免“任务中心有数据但结果页为空” |
| MIG-04 | 重启后的业务持久化 | 已创建用户、数据集、算法、指标、Run、社区资源 | 重启后端与前端，重新登录后查看各列表并执行 1 次新 Run | 核心业务记录仍在，且新任务可继续创建 | 证明系统不是仅靠内存态存活 |
| MIG-05 | 关闭 Redis fallback 的核心回归 | SQL 主存储已完整接入 | 设置 `ABP_SQL_FALLBACK_REDIS=0` 后重启，回归登录、数据集、算法接入、Run、结果对比 | 核心业务链路仍可正常运行 | 可作为 MySQL 主持久化的答辩证据 |

### 10.2 算法接入与社区联动专项

| 编号 | 场景 | 前置条件 | 操作步骤 | 预期结果 | 稳定性关注点 |
|---|---|---|---|---|---|
| MIG-06 | 社区下载算法后可运行 | 用户 A 已发布接入算法到社区；用户 B 已登录 | 用户 B 下载算法 -> 打开发起测评 -> 选择该算法创建 Run | 下载副本能在 `NewRun` 中出现并真实运行成功 | 不应出现“社区页显示已下载但不可运行” |
| MIG-07 | 接入算法发布社区后不产生 `#2` | 用户已有 1 条审核通过的接入申请 | 在“算法接入”或“算法库”发布到社区，再刷新算法库与发起测评 | 用户运行副本与社区副本名称稳定，不出现 `#2` | 同一 submission 双副本不应互相挤出重名 |
| MIG-08 | 接入算法三态联动 | 同一条接入申请已具备用户副本、社区副本，可进一步被平台收录 | 依次验证“仅用户可用”->“发布社区”->“管理员收录平台”三阶段 | 用户算法、社区算法、平台算法展示口径一致，且不重复显示同源资源 | 平台收录后旧副本应按设计清理 |
| MIG-09 | 社区算法下载后不挤掉本人资源 | 用户 A 发布 2 条算法到社区；用户 B 已有本人接入算法 | 用户 B 连续下载多条社区算法，再查看“算法库”“社区中心”“发起测评” | 本人算法、下载算法、社区状态都应同时正确显示 | 不应出现“下载一个后另一个消失” |
| MIG-10 | 发起测评的同源去重正确 | 同时存在平台算法、本人接入算法、社区下载副本 | 打开发起测评，观察同一来源资源是否重复显示 | 同源资源只显示一次，不同来源资源应保留 | 不能因为去重把可运行下载副本误折叠掉 |
