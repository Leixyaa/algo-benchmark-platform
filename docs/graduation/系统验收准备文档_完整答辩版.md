## 一、系统演示流程（6～12 分钟）

### 1.1 演示前准备（勾选清单）

- 启动 **Redis**（Celery Broker / 部分缓存依赖）
- 启动 **后端 API**：`uvicorn backend.app.main:app`（或项目约定端口）
- 启动 **Celery Worker**：`celery -A backend.app.celery_app:celery_app worker ...`（与 `tasks.execute_run` 对应）
- 启动 **前端**：在 `web/` 下 `npm install` → `npm run dev`；交付/验收构建执行 `npm run build`
- **桌面端**（若演示）：先 `cd web && npm run build:desktop`，Electron/脚本加载 `web/dist-desktop`（参见 `docs/desktop-quickstart.md`）
- 准备账号：**管理员** + **普通用户**（各至少一个）
- 在仓库根目录执行自检：`powershell -ExecutionPolicy Bypass -File .\scripts\demo_preflight_check.ps1`
- 对照固化数据：`docs/graduation/演示数据固化清单.md`
- **说明**：停止 `uvicorn` 时 Ctrl+C 出现 `KeyboardInterrupt` / `CancelledError` 为正常关机输出，**不是**业务 Bug。

---

### 1.2 演示步骤（建议顺序）

#### A. 用户端主链路（约 3～5 分钟）

**操作流程**：

1. 打开「**数据集**」：确认目标任务下 **配对数 / pairs_by_task** 合理（>0 或符合演示口径）  
2. 打开「**算法库**」：展示平台内置算法与用户算法；可点到「算法接入 / 接入申请记录」一带而过  
3. 打开「**发起评测**」：  
   - 第一步：任务类型、数据集、指标、（可选）评测模式等  
   - 第二步：**多选算法**（强调支持批量）、可选 **保存/加载参数预设**（含多算法 `algorithm_ids`）  
   - 启动 Run  
4. 打开「**任务中心**」：展示排队/运行/完成、**详情**中的时间链路与指标；可演示 **取消 / 删除 / 批量清除**（终态任务）  
5. 打开「**结果对比**」：筛选、排序、推荐说明、**导出**（CSV/XLSX/Markdown 等以实际页面为准）

**讲解要点（话术提纲）**：

- 完整闭环：**数据集 → 算法 → Run（异步）→ 指标 → 对比与导出**  
- **图像 + 视频**任务类型均已纳入主线（具体任务键见下文「功能模块」）  
- 发起评测为 **两步向导**；运行参数以各算法 **默认参数** 为准（避免过期「参数方案」导致不一致）  
- **参数预设**可保存 **多算法集合**，加载后一键回填  
- **真实执行链路**：Run 经 Celery 进入 `execute_run`，算法在隔离环境中执行，再按所选指标计算结果——**非页面模拟分数**

---

#### B. 管理端治理链路（约 2～4 分钟）

**操作流程**：

1. 切换 **管理员** 进入「**管理后台**」  
2. 演示：**算法接入审核**、**指标审核**、（可选）**举报处理**  
3. 演示社区资源 **下架 / 删除** 后的联动（通知、列表变化等）  
4. （若版本已包含）「**注册用户**」列表中的 **删除用户**（测试账号清理）

**讲解要点**：

- 自定义算法/指标进入运行链路前需 **审核 + runtime_ready** 等门禁（以页面与接口为准）  
- **社区治理闭环**：审核、下架、举报、评论管理等  
- **历史记录后端化**：算法接入历史、指标提交历史存于独立表（`abp_algorithm_history` / `abp_metric_history`），与原资源是否仍存在 **解耦**，便于验收讲解「删除资源不等于删除历史留痕」

---

#### C. 社区与资源流转（约 1～2 分钟，可与 A/B 合并）

**操作流程**：社区中心浏览 → 资源详情 → 下载到个人库 → 回到「我的算法/数据集」使用  

**讲解要点**：下载副本、来源信息、与评测主线的衔接。

---

#### D. 通知与个人信息（约 1 分钟）

**操作流程**：右上角「**我的通知**」→ 筛选/已读/清理；用户下拉「**个人信息**」→ 修改 **显示名称**（不等于用户名）

**讲解要点**：站内通知用于下架、举报结果等；显示名称用于展示友好。

---

#### E. AI 智能客服（约 1 分钟，可选）

**操作流程**：登录后右下角 **AI 客服**（游客通常禁用）→ 提问「如何发起评测」「算法如何接入」等  

**讲解要点**：FAQ 命中稳定回答；未配置外部模型时有 **离线兜底**；管理员可控制是否展示参考文档来源。

## 三、核心逻辑讲解（后端链路版）

> 本节不讲具体函数实现细节，只讲完整业务链路。答辩时按“谁触发 -> 系统做什么 -> 结果如何回到页面”来讲。

---

### 3.1 评测主链路：从点击“发起评测”到任务完成

1. 用户在前端提交任务后，后端先做参数与资源合法性校验（任务类型、数据集、算法、指标是否可运行）。  
2. 校验通过后先落一条 `queued` 任务记录，确保“先有记录再执行”，页面能立刻看到任务。  
3. 后端把任务交给异步执行器（Celery Worker），避免 HTTP 请求被长任务阻塞。  
4. Worker 执行时更新状态为 `running`，按任务类型读取样本并调用算法运行时。  
5. 算法输出完成后计算指标并汇总，写回最终状态 `done`；异常则写 `failed` 和错误信息。  
6. 前端任务中心通过列表刷新看到状态流转，实现“创建即见、过程可查、结果可追溯”。

---

### 3.2 算法运行链路：为什么是“真实执行”而不是模拟分数

1. 平台把待评测样本交给算法运行层，不在 API 进程里直接执行用户代码。  
2. 算法以子进程方式运行，主进程只负责调度、超时控制、结果采集。  
3. 子进程输出结果后，系统再按指标定义计算分数。  
4. 这样可以同时保证隔离性（用户代码不污染主服务）和真实性（分数来自真实输出）。

---

### 3.3 存储链路：Redis + SQL 为什么要协同

1. 关键业务记录同时落持久化存储与缓存存储。  
2. 读取时做合并与择优，避免单侧异常导致“页面看不到真实数据”。  
3. 列表接口以合并结果返回，提升演示与生产场景的鲁棒性。  
4. 对用户来说表现为：刷新后状态更稳定，不容易出现“刚提交就丢任务”的错觉。

---

### 3.4 历史记录链路：为什么资源删了历史还在

1. 算法接入历史、指标提交历史采用独立历史数据模型。  
2. 历史记录保存的是“当时的提交快照”，不是主资源对象的引用。  
3. 即使资源被下架或删除，历史仍可查询、下载、审计。  
4. 这对验收/答辩很重要：可以解释“业务治理”和“留痕追溯”同时成立。

---

### 3.5 数据集可用性链路：为什么有时不让创建任务

1. 数据集导入后会做目录扫描与配对统计。  
2. 平台按任务类型计算可用样本数（如 `pairs_by_task`）。  
3. 发起评测时若当前任务可用样本为 0，会阻止创建并提示先修复数据集。  
4. 这保证了评测不是“空跑”，也避免了无意义结果。

---

### 3.6 社区治理链路：发布、举报、下架如何闭环

1. 用户可发布资源到社区并被他人浏览、下载、评论。  
2. 管理员可处理举报、删除违规评论、下架资源。  
3. 治理动作会同步影响社区可见性与相关通知。  
4. 目标是形成“开放发布 + 可追责治理”的完整机制。

---

### 3.7 AI 客服链路：为什么回答会“像平台内客服”

1. AI 聊天请求会携带当前页面上下文（路径、标题、用户角色）。  
2. 后端先从平台文档中检索相关片段，再组织回答。  
3. 有外部模型时走检索增强回答；无外部模型时走本地兜底。  
4. 管理员可查看回答引用来源，用于验证“回答依据来自平台文档”。

## 四、核心逻辑讲解（前端链路版）

### 4.1 发起评测链路（用户最常走）

1. 第一步确定任务类型与数据集。  
2. 第二步选择算法与指标，支持单算法或多算法批量创建。  
3. 前端会过滤不可运行资源，尽量避免把失效配置提交给后端。  
4. 创建成功后跳转或留在当前页，失败会给出可操作提示（不是只有“失败”两个字）。

---

### 4.2 任务中心链路（状态可见、可控）

1. 页面展示任务从排队到完成/失败的状态变化。  
2. 用户可查看详情、取消、删除、批量清理终态任务。  
3. 前端以服务端状态为准，避免本地假状态长期滞留。  
4. 这样用户能明确看到“任务是否真的执行、失败在哪一步”。

---

### 4.3 结果对比链路（结果可解释）

1. 结果页按任务与数据集聚合同口径 Run。  
2. 指标选择遵循“内置指标常驻 + 自定义指标按任务可用性筛选”。  
3. 排序、推荐、导出使用同一份对比数据，保证口径一致。  
4. 用户导出的内容与页面看到的排序结果一致。

---

### 4.4 社区与资源流转链路（从“看见”到“可用”）

1. 用户在社区浏览公开资源并查看详情。  
2. 下载后资源进入个人库，再参与发起评测。  
3. 社区治理动作会反馈到列表可见性，避免“已下架还可继续使用社区副本”的错觉。  
4. 这条链路体现平台不仅“能评测”，也“能协作分享”。

---

### 4.5 管理后台链路（审核与治理）

1. 管理端集中处理审核、举报、下架、评论治理。  
2. 操作后前端会局部刷新，保证结果即时可见。  
3. 管理功能与普通用户页面形成职责分离，避免权限混用。  
4. 从验收角度看，这是平台“可运营”的关键能力。

## 六、技术问题答辩（高频 Q&A）

### 架构与执行

**Q1：整体架构怎么描述？**  
**A：** 浏览器/Electron → FastAPI → Redis/Celery Worker 执行 Run → 结果写回存储；前端通过 REST 拉取列表与详情。

**Q2：怎么证明不是「假跑分」？**  
**A：** 展示 `tasks.execute_run` + `algorithm_runtime` 子进程调用链；同一 Run 可复查日志与输出字段。

**Q3：视频评测是逐帧还是只采一帧？**  
**A：** 以当前 `tasks.py` 实现为准；答辩现场可打开对应分支简述「逐帧/分段聚合」策略。

---

### 数据与安全

**Q4：用户密码怎么存？**  
**A：** 使用哈希存储（具体算法见 `backend/app/auth.py`），验收勿演示明文密码。

**Q5：JWT 存在哪？**  
**A：** 前端常见为 `localStorage`/`sessionStorage` 保存 token（以实际前端为准），请求头 `Authorization: Bearer ...`。

---

### 性能与稳定性

**Q6：高并发下会不会拖垮 API？**  
**A：** 重活不在 API 进程内完成，而在 Worker；API 主要负责鉴权与调度。

**Q7：Redis 挂了怎么办？**  
**A：** 依部署策略：开发环境可能降级；生产应保证 Redis 高可用。可诚实说明当前项目定位是「评测平台/演示」。

---

### 功能与产品

**Q8：删除算法后历史 Run 还在吗？**  
**A：** Run 记录独立存；页面可能显示「关联资源已删除」类提示，但记录仍在（具体以前端展示为准）。

**Q9：历史记录为什么不用浏览器本地存？**  
**A：** 换设备/清缓存会丢；后端表持久化才能覆盖 Web 与桌面端一致体验。
